# SyncWire — Android

The Android half of the [SyncWire](../projects/syncwire-server) notification relay. Runs as a long-lived `NotificationListenerService` that forwards every system notification to the SyncWire server.

> **Status:** **M1** complete. Forwarding works end-to-end against the server. Auth (Auth0), offline outbox + retry, and `dataSync` foreground service are scoped for the next milestones.

---

## What it does

- Lives as a `NotificationListenerService` in the system
- Receives every posted notification (WhatsApp, Gmail, Slack, anything user-facing)
- POSTs each one to the configured SyncWire server as JSON
- Dedupe + persistence happens server-side; the app just forwards

The app is **content-agnostic** — the server stores whatever the app sends, including custom `sourceType` tags for future integrations.

## Build

Prerequisites: JDK 17+ (we tested with JDK 26), Android SDK with `platforms;android-36` installed.

```bash
# One-time
sdkmanager "platforms;android-36" "build-tools;36.0.0"

# Build
./gradlew :app:assembleDebug
# → app/build/outputs/apk/debug/app-debug.apk
```

The debug APK is unsigned (debug keystore is auto-generated by AGP on first build).

## Install + run

```bash
adb install -r app/build/outputs/apk/debug/app-debug.apk
adb shell am start -n com.example.syncwire/.MainActivity
```

In the app:
1. **Server URL** — defaults to `http://10.0.2.2:18080/api` (the emulator's host loopback). For a real device on the same Wi-Fi, use your machine's LAN IP, e.g. `http://192.168.1.10:18080/api`.
2. Tap **Open settings** → toggle SyncWire **on** under "Notification access" → back to the app.
3. Tap **Check** — the "Notification access" card should flip to "Enabled".
4. Tap **Send test** to POST a synthetic notification, or trigger any real one (WhatsApp, Gmail, etc.).

Verify on the server:

```bash
curl "http://127.0.0.1:18080/api/notifications?deviceId=<id_shown_in_app>"
# or
docker compose -C /path/to/syncwire-server exec postgres \
  psql -U syncwire -d syncwire \
  -c "SELECT id, source_type, sender, content, package_name, received_at \
      FROM notifications ORDER BY received_at DESC LIMIT 5;"
```

## How it works

1. User grants **Notification Access** to the app in system Settings. The OS binds `SyncwireListenerService` and keeps it alive.
2. On every `onNotificationPosted` callback, the service builds a `NotificationPayload` (id, deviceId, sourceType, sender, content, packageName, timestamp) and POSTs it as JSON to `{serverUrl}/notifications`.
3. The server stores the row, dedupes by `id`, and returns 201. The app does not retry on failure in M1 — failures are logged and dropped. (M3 will add a Room outbox.)
4. The Compose UI's "Refresh list" calls `GET /api/notifications?deviceId=...` and renders the rows.

`deviceId` is a UUID generated on first launch and stored in DataStore. M2 will replace this with a server-issued id + api key.
